# SaSHA VM

The SaSHA VM is broken into layers:
- The loader loads all of the 8-bit information into the memory, executes it, and clears it.
- The executor takes all of the memory and runs opcodes on it, handles stack operations etc.

The SaSHA VM has a program segment that starts at `0x00`, while the data segment starts at `0x0500`.
**If segments override, then the program no longer is allowed to run.**

The function segment is generated by the bytecode generators for SVL and the assembly language,
as it is determined by where a program concludes with a `HALT` instruction.

Finally, the SaSHA VM is `64KB` of storage, meaning the max program size is aroud 12.5 KB.

## Segmentation `SegmentAllocator`

The segment allocator class manages the length of data segments, and will error out the program if there is an overlap.

Think of the layout of a SA executable kind of like a map:
```
.CODE 0000000000000000000000 .DATA 00000000000000000000000000000
```

The `code` and `data` segments have a strict amount of bytes they are allowed before they intersect. And if code intersects data (data can't move backwards, every `data` operation should occur **with an offset of wherever it's defined**) then the program has a `MAV` (Memory access violation) and needs to exit prematurely.

Think of the `SegmentAllocator` class as the internal critic, it is used to define sizes of blocks, block overriding, and to manage the actual total allocation of the program.
